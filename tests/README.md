# テスト

このドキュメントは初期のテスト方針を記録します。

最終的なテスト構造は cc-sdd がプロジェクトタスクを生成した後に精緻化します。

## テストカテゴリ

期待するカテゴリ：

```text
unit
integration_fake
cloud
e2e_cloud
```

## ユニットテスト

外部サービスなし。

期待するテスト対象：

- スキャナーファイル名のパース
- スキャンタイムスタンプの抽出
- 日本の日付の正規化
- 元号のフォーマット
- ファイル名の生成
- ファイル名のサニタイズ
- 重複サフィックスロジック
- 要確認/エラープレフィックスの処理

## フェイク統合テスト

外部サービスなし。

以下のフェイクアダプターを使用：

- Drive
- OCR
- LLM 抽出
- ログ/通知

クラウド認証情報なしでアプリケーションフローをテストするためのものです。

## クラウド統合テスト

Google Cloud サービスを呼び出すことができますが、ホスト側 GCP Test Broker 経由のみです。

Claude Code はこれらのテストを実行することが許可されています。

DevContainer は GCP 認証情報を受け取ってはいけません。

## E2E クラウドテスト

初期実装より後で任意に実施します。

専用の Google Drive テストフォルダを使用することがあります。

以下を検証します：

- アップロード/一覧取得
- PDF 解析
- 生成されたファイル名
- リネーム
- クリーンアップ

## マーカーポリシー

誤ったクラウド呼び出しを避けるために pytest マーカーを使用します。

概念的なコマンド：

```bash
pytest -m "not cloud and not e2e_cloud"
pytest -m cloud --run-cloud-tests
pytest -m e2e_cloud --run-e2e-cloud-tests
```

正確なコマンドは実装中に決定します。
