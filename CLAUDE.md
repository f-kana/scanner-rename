# CLAUDE.md

Brotherスキャナーのスキャン PDF を自動リネームするシステム。プロジェクトの詳細は `docs/initial-context.md`、セキュリティ方針は `docs/security-notes.md`、アーキテクチャ決定は `docs/adr-candidates/` を参照。

## 重要: まず cc-sdd を実行する

このプロジェクトは cc-sdd を使って開発することを前提としています。

cc-sdd フローが生成するまで、最終的な `spec.md`、`design.md`、`tasks.md` を手動で作成したり、過度に詳細化したりしないでください。

以下のドキュメントは cc-sdd への事前インプット素材としてのみ使用します：

- `docs/initial-context.md`
- `docs/security-notes.md`
- `docs/adr-candidates/*.md`
- `docs/product-backlog.md`

cc-sdd が正規の仕様/設計/タスクを生成した後は、生成された cc-sdd 成果物を正とします。矛盾が生じた場合は、コアな挙動を変更する前にユーザーに確認してください。

## シークレットと認証情報

認証情報やトークンを読み取り、印刷、コピー、または外部に持ち出してはいけません。クラウド統合テストはホスト側 GCP Test Broker 経由のみで実行します。詳細なルールは `.claude/rules/security.md` を参照。

## ランタイム LLM プロンプト

`app_llm_prompts/` は、アプリケーションが実行時に Gemini または他の文書解析 LLM に対して使用するプロンプトのためのディレクトリです。Claude Code の指示のためのものではありません。

## テスト方針

pytest マーカーでテストを分類します（`unit`, `integration_fake`, `cloud`, `e2e_cloud`）。デフォルトのローカルテストは外部サービスを必要としないようにします。詳細は `tests/README.md` を参照。

## 開発スタイル

小さくテスト可能なステップを優先します。

設定ファイル（`.gitignore`、linter設定、CI設定など）やツールの利用パターンについては、公式テンプレートやドキュメントの推奨方法を先に調査し、それに従います。独自のカスタムパターンを作る前に、標準的なアプローチが存在しないか確認してください。

プロジェクトの設定やルールの変更はこのリポジトリ内で完結させます。`~/.claude/` など、リポジトリ外のユーザースコープ設定を変更しないでください。

Python の関数の引数と戻り値には型ヒントを使用します。

ドメインロジックをインフラアダプターから分離します。

## 時期尚早な実装をしない

cc-sdd がタスクを生成するまで、明示的に求められない限り以下を実装しないでください：

- 最終的な本番 Cloud Run デプロイメント
- Terraform/IaC
- Drive の完全な E2E
- Firestore またはその他のステートストア
- Slack/Gmail への直接通知
- `.claude/skills/`

## その他の考慮事項

- Promptingに音声入力を使うことがあるため、プロンプトが誤変換を多く含むケースがある。誤変換であることを前提にプロンプトを解釈せよ。
