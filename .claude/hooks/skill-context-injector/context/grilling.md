## 提案前の事実確認

質問や選択肢を提示する前に、外部ツール・ライブラリ・feature の機能を前提とする場合は、公式ドキュメントで実現可能性を確認してください。

確認が必要な例：
- DevContainer features の機能（`https://github.com/devcontainers/features` で確認）
- npm/Python パッケージの機能（公式ドキュメント、README、PyPI で確認）
- クラウドサービス API の機能（公式ドキュメントで確認）
- 「このツールには〇〇が含まれているはず」という推測をしている場合
- 新しい名前（スキル名、ブランチ名等）を提案する場合、プロジェクトの命名規則を事前に確認する

## Build vs Buy の早期判断

実装手段を議論する前に、既存の公開スキル・プラグイン・ライブラリで同等のものがないか調査する。特に:

- APM で導入可能なスキル（`apm.yml` の既存依存関係も確認）
- Claude Code Plugin マーケットプレイスの公開プラグイン
- 著名なハーネス（Superpowers 等）が提供する関連スキル

「自作する」という判断は、既存ソリューションの不足を確認した後に行う。

WebFetch、Read、Bash で事実確認してから、質問や推奨案を提示してください。

## 事前確認

スキルの本題に入る**前に**、ユーザーに以下を確認してください：

> AIに聞く前に、ご自身でこのテーマについて考えを整理しましたか？現時点でのお考えを簡潔に教えてください。

ユーザーが自分の考えを述べるまで、スキルの本題に入らないでください。
