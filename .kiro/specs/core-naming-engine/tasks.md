# Implementation Plan: core-naming-engine

- [x] 1. Foundation: ドメインパッケージ基盤の整備
- [x] 1.1 domain サブパッケージの骨格とドメイン例外階層を作成する
  - `src/scanner_rename/domain/` と `tests/unit/` のパッケージ構造を新設する
  - 失敗を型で区別する例外階層（基底・元号範囲外・命名入力不備）を定義する
  - 例外の継承関係と型を検証する `unit` マーカー付きテストを追加する
  - 完了条件: デフォルトのテスト実行（外部サービスなし）で新規 unit テストが収集・成功し、ドメイン例外を import できる
  - _Requirements: 6.3, 6.4_

- [x] 2. スキャナー生成ファイル名のパースと検証
- [x] 2.1 スキャナー生成名を構造化された値として解釈・復元できるようにする
  - パターン `^\d{14}_\d{3}\.pdf$` の判定とタイムスタンプ（YYYYMMDDhhmmss）の日時変換を実装する
  - パターン不一致およびカレンダー上無効な日時（月 13・日 32・時 25 等）は「該当しない」として扱う
  - パース結果から元のファイル名文字列を正確に復元できるようにする
  - 完了条件: 合意例 `20260507132742_001.pdf` のパース、ラウンドトリップ、無効ケースのパラメタライズ unit テストがすべて成功する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_

- [x] 3. 状態プレフィックスの付与・除去・分類
- [x] 3.1 ファイル名による処理状態の表現と手動リトライのラウンドトリップを実現する
  - `_needs_review_` / `rename_error_` プレフィックスの付与・除去を実装する
  - 任意のファイル名を「未処理スキャナー生成名 / 要確認 / エラー / 処理対象外」の 4 値に分類する（例外を送出しない）
  - プレフィックス除去後がスキャナー生成名パターンに一致しない名前は処理対象外とする
  - 完了条件: 付与→除去で元のスキャナー生成名に戻るラウンドトリップと 4 値分類の境界ケースの unit テストが成功する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5_

- [x] 4. 日付処理と元号変換
- [x] 4.1 (P) 元号テーブルと西暦⇔元号の相互変換を実装する
  - 明治・大正・昭和・平成・令和の改元日を持つ静的テーブルを定義する（改元日当日は新元号）
  - 西暦→元号+元号年、元号+年月日→西暦の双方向変換を実装する
  - 対応表の範囲外の日付は変換不能として明示的に通知する
  - 完了条件: 各改元日の当日・前日の境界パラメタライズと双方向変換の整合を検証する unit テストが成功する
  - _Requirements: 3.1, 3.2, 3.3, 3.6_
  - _Boundary: japanese_era_
- [x] 4.2 ファイル名用の日付コンポーネント整形を実装する
  - 元号表記由来の日付は `YYYYMMDD(<元号略号><元号年>)` 形式（例: `20211001(R3)`）で整形する
  - 西暦のみ表記の日付は元号を付与せず `YYYYMMDD` 形式で整形する
  - 完了条件: `20211001(R3)` の完全一致生成と西暦のみ整形の unit テストが成功する
  - _Requirements: 3.4, 3.5_

- [x] 5. ファイル名サニタイズと重複解決
- [x] 5.1 (P) ファイル名サニタイズを実装する
  - ファイルシステムで問題となる文字（パス区切り・制御文字等）を安全な文字に置換し、連続する区切りの余剰を畳む
  - 先頭・末尾の空白とピリオドを除去し、冪等に動作させる
  - 上限長（200 文字）超過時にタイトル部分を優先的に短縮できるよう、コンポーネント単位と全体の 2 段の整形を提供する
  - 完了条件: 禁止文字置換・トリム・冪等性・長さ上限の unit テストが成功し、日本語（全角）文字が無加工で通る
  - _Requirements: 5.4, 5.5, 5.6_
  - _Boundary: sanitize_
- [x] 5.2 (P) 重複サフィックス解決を実装する
  - 呼び出し側が渡す既存ファイル名一覧に対し、同名が存在する場合は拡張子直前に `_2`, `_3`, ... と最小の未使用番号を挿入する
  - 同名が存在しない場合は候補名をそのまま返す
  - 完了条件: initial-context.md の重複例（`_2`, `_3` の連番）を再現する unit テストが成功する
  - _Requirements: 5.1, 5.2, 5.3_
  - _Boundary: dedup_

- [x] 6. 命名エンジン
- [x] 6.1 命名入力の契約型と入力検証を定義する
  - タイトル・文書日付・元号表記フラグ・期間・発行元からなる不変の命名入力型と、期間種別（年分・年度分・明示的期間）の型を定義する
  - タイトルが空・空白のみ、または期間種別に必須のフィールドが欠落している場合は入力不備として明示的に通知する
  - 完了条件: 型の不変性（frozen）と入力不備ケースの unit テストが成功する
  - _Requirements: 4.8_
  - _Boundary: naming_
- [x] 6.2 メタデータからのファイル名組み立てを実装する
  - `<文書日付>_<期間>_<タイトル>_<発行元>.pdf` の順で `_` 連結し、期間なし・発行元なしのコンポーネントは省略する
  - 期間を `YYYY年分` / `YYYY年度分(YYYYMM-YYYYMM)` / `YYYYMM-YYYYMM` の各形式で整形する
  - 文書日付がない場合はスキャンタイムスタンプの日付部分にフォールバックする
  - 各コンポーネントをサニタイズしてから連結し、長さ超過時はタイトルを優先的に短縮する
  - 完了条件: initial-context.md の合意例 3 件（住宅ローン証明書・確定申告書・医療費通知）の完全一致再現、省略規則の組み合わせ、`対象` を含む固定文字列を生成しないことの unit テストが成功する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7, 4.9_
  - _Depends: 4.2, 5.1_
  - _Boundary: naming_

- [ ] 7. 統合と全体検証
- [x] 7.1 公開 API の再エクスポートと純粋性の検証を行う
  - 消費者（extraction-pipeline）が単一の入口から import できるよう、domain パッケージの公開 API を再エクスポートする
  - domain 配下のモジュールが標準ライブラリ以外を import しないことを検証する unit テストを追加する
  - 代表的な入力に対する決定性（同一入力→同一出力）の検証を追加する
  - 完了条件: 公開 API 経由の import で全機能が利用でき、純粋性・決定性テストが成功する
  - _Requirements: 6.1, 6.2_
- [ ] 7.2 全体の品質ゲートを通過させる
  - デフォルト設定でのテスト全実行（外部サービスなしで完走）を確認する
  - lint・型検査を全ファイルで通過させる
  - 要件書の全受入基準がいずれかの unit テストで検証されていることを突き合わせ、漏れがあればテストを補完する
  - 完了条件: テスト・lint・型検査がすべて成功し、受入基準とテストの対応が揃っている
  - _Requirements: 6.3, 6.4_
