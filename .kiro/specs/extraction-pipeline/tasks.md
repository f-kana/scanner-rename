# Implementation Plan: extraction-pipeline

前提: core-naming-engine の実装タスク（`src/scanner_rename/domain/` の公開 API）が完了していること。本プランの全タスクが domain API（`classify_filename`, `build_filename`, `resolve_duplicate`, `with_state_prefix` 等）を消費する。

- [ ] 1. 基盤: 抽出スキーマとポート契約の確立
- [ ] 1.1 Gemini 構造化出力スキーマの型定義
  - 文書日付（元号シグナル付き）・書類タイトル・発行元・文書種別・期間（年分/年度分/明示期間の単一判別型）・理由を、値 + 信頼度 + エビデンスの不変値オブジェクトとして定義する
  - 信頼度の範囲（0–1）と期間種別ごとの必須フィールドの不変条件検証を実装する
  - unit テストで不変条件違反の拒否と正常構築（欠落フィールドの None 表現を含む）が検証され、テストが通っている
  - _Requirements: 2.2, 2.3, 2.5, 2.6_
- [ ] 1.2 3 ポート契約と共通例外階層の定義
  - 監視フォルダにスコープされたファイル保管の抽象（一覧・内容取得・リネーム）、OCR の抽象、構造化抽出の抽象と、各 DTO を Protocol + frozen dataclass で定義する
  - ポート失敗の共通基底例外と 3 派生を定義し、正常応答と失敗の表現を契約として確定する
  - 契約面が再エクスポートモジュールから import でき、Pyright の型検査が通っている
  - _Requirements: 1.1, 1.4, 2.1, 7.1_

- [ ] 2. ランタイムプロンプト素材とローダ
- [ ] 2.1 抽出・命名ポリシーの確定版作成
  - 抽出ポリシーのドラフトを、確定スキーマ（期間の単一判別構造、フィールドごとの信頼度・短いエビデンス）と整合する構造化データ記述および日付優先順位・元号の扱いを含む確定版に改稿・改名する
  - 命名ポリシーのドラフトを確定版に改稿・改名し、機械的な安全規則（サニタイズ・重複・長さ・状態プレフィックス）を含めない境界を維持する
  - `app_llm_prompts/` にドラフト 2 ファイルが残らず確定版 2 ファイルと更新済み README が存在している
  - _Requirements: 2.4, 6.1, 6.2, 6.5_
  - _Depends: 1.1_
- [ ] 2.2 プロンプトローダの実装
  - 確定版 2 ファイルを読み込んで非空を検証し、欠落・空・読み込み不能を明示的なエラー（対象パス入り）として送出する
  - unit テストで正常読み込み（リポジトリの確定版 2 ファイルを含む）と欠落・空ファイルのエラーが検証され、テストが通っている
  - _Requirements: 6.1, 6.2, 6.3, 6.4_

- [ ] 3. フェイクアダプタ（ポート契約の参照実装）
- [ ] 3.1 (P) フェイク Drive の実装
  - インメモリのファイル集合に対する一覧（決定的順序）・内容取得・リネームを提供する
  - 未知のファイル ID と故障注入（指定ファイル・指定操作での失敗）が契約どおりの例外型で表現される
  - unit テストで契約適合（リネーム後の一覧反映・例外型・決定的順序）が検証され、テストが通っている
  - _Requirements: 7.2, 7.3_
  - _Boundary: adapters/fake FakeDrive_
- [ ] 3.2 (P) フェイク OCR とフェイク抽出の実装
  - 登録済みデータに基づく決定論的な OCR 応答・構造化抽出応答を提供する
  - 未登録入力と故障注入が契約どおりの例外型で表現される
  - unit テストで両フェイクの契約適合が検証され、テストが通っている
  - _Requirements: 7.2, 7.3_
  - _Boundary: adapters/fake FakeOcr FakeExtraction_

- [ ] 4. アプリケーション層（命名決定とフロー実装）
- [ ] 4.1 設定と命名決定マッパーの実装
  - 信頼度閾値を持つ設定値オブジェクト（範囲検証付き、デフォルト 0.7）を定義する
  - 信頼度ゲートを実装する: タイトル欠落・閾値未満は要確認判定（フィールド名と信頼度を含む理由付き）、文書日付の欠落・閾値未満はスキャンタイムスタンプへフォールバック、発行元・期間の欠落・閾値未満は省略
  - 抽出スキーマから命名入力への射影（元号シグナル・期間 3 種の変換を含む）を実装する
  - unit テストで閾値境界（ちょうど採用・直下不採用）、フォールバック、省略、期間変換、要確認理由が検証され、テストが通っている
  - _Requirements: 3.1, 3.2, 3.3, 4.1, 4.2, 4.3_
- [ ] 4.2 リネームパイプラインの成功パス実装
  - 監視フォルダ直下の一覧取得 → ファイル名分類 → 未処理スキャナー生成名のみ処理対象とし、それ以外は変更せずスキップ記録する（0 件時は何も変更せず要約を返す）
  - 処理対象ごとに内容取得 → OCR → 構造化抽出 → 命名決定 → ファイル名生成 → 重複解決（初回一覧 + 本実行で確定した名前の集合に対して）→ リネームを実行する
  - ファイルごとの結果（成功・スキップ）と理由を含む実行要約を返す
  - フェイク 3 種を注入したパイプラインが、スキャナー生成名の PDF を最終名にリネームし要約を返すことが確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 3.4, 3.5, 3.6, 5.4_
- [ ] 4.3 要確認・エラー遷移と障害分離の実装
  - 要確認判定のファイルを `_needs_review_` プレフィックス名にリネームし、理由を要約に記録する（技術エラーと区別する）
  - ファイル単位の try 境界でポート例外・ドメイン例外・予期しない例外を捕捉し、`rename_error_` プレフィックス名へのリネームと例外要約（型名含む、認証情報・OCR 全文なし）の記録に変換する
  - エラー遷移のリネーム自体が失敗した場合は記録のみ行い、残りのファイルの処理を継続する
  - 故障注入フェイクを使い、1 ファイルの失敗後もバッチが継続し全ファイルの結果が要約に含まれることが確認できる
  - _Requirements: 4.1, 4.3, 4.4, 5.1, 5.2, 5.3, 5.5_

- [ ] 5. フェイク統合テストによるフロー全体検証
- [ ] 5.1 成功フローの integration_fake テスト
  - フェイク一式とパイプラインを組み立てる共有フィクスチャ（conftest）を整備する
  - 合意済み 3 例（住宅ローン証明書・確定申告書・医療費通知）をスキャナー生成名から最終名まで完全一致で再現する（元号表記・年分・年度分を含む）
  - 処理済み名・要確認・エラー・対象外ファイルが変更されないこと、0 件時の正常終了、既存重複とバッチ内重複のサフィックス解決を検証する
  - `pytest -m integration_fake` が外部サービスなしで通っている
  - _Requirements: 1.3, 1.5, 2.1, 2.2, 2.6, 3.1, 3.4, 3.5, 3.6, 7.4_
- [ ] 5.2 失敗・回復シナリオの integration_fake テスト
  - タイトル低信頼度の文書が `_needs_review_` 元名に遷移し理由が要約に記録されることを検証する
  - OCR 故障・抽出故障・リネーム故障それぞれの `rename_error_` 遷移、混在バッチ（成功 + 要確認 + エラー）での障害分離、エラー遷移自体の失敗時の継続を検証する
  - 要確認・エラーのファイル名を元に戻した状態での再実行が再処理につながること（手動リトライ）を検証する
  - デフォルトのローカルテスト実行（`unit` + `integration_fake`）が外部サービスへ接続せずに完走している
  - _Requirements: 4.1, 4.3, 4.4, 5.1, 5.2, 5.3, 5.4, 5.6, 7.4, 7.5_
