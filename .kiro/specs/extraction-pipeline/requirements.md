# Requirements Document

## Project Description (Input)

PDF を検出してから Drive 上でリネームするまでのアプリケーションフロー全体（検索 → OCR → LLM 抽出 → 命名 → リネーム → 状態遷移）がまだ存在しない。また、外部サービス（Drive / Document AI / Gemini）に依存せずにこのフローを検証する手段がない。

core-naming-engine（純ドメイン）が別スペックで定義済み。ポート定義・アプリケーションサービス・フェイクアダプタは未着手。Gemini への抽出指示素材は `app_llm_prompts/*.draft.md` にドラフトのみ存在する。

Drive / OCR / LLM 抽出の 3 ポートを定義し、フェイクアダプタで差し替え可能にする。アプリケーションフロー全体（`/From_BrotherDevice` 直下のパターン一致 PDF を列挙 → OCR → 構造化抽出 → 命名 → リネーム、失敗時は `_needs_review_` / `rename_error_` 遷移）を実装する。Gemini 構造化出力のスキーマを入出力型として定義し、抽出・命名ポリシーのランタイムプロンプト素材を `app_llm_prompts/` で確定版にする。アプリレベルのエラーハンドリングポリシー（低信頼度と技術エラーの分岐、1 ファイルの失敗が他ファイルの処理を止めない）を定義し、pytest `integration_fake` でフロー全体を外部サービスなしで検証できるようにする。

## Introduction

本機能 extraction-pipeline は、スキャン PDF 自動リネームシステムのアプリケーション層である。監視フォルダ直下の未処理スキャン PDF を検出し、OCR と構造化抽出で命名に必要なメタデータを取得し、core-naming-engine の命名エンジンで人間可読なファイル名を生成して Drive 上でリネームする。抽出が不確かな文書は要確認として、技術的に失敗した文書はエラーとしてファイル名プレフィックスでマークし、いずれの場合も他のファイルの処理を止めない。外部サービスへのアクセスはすべて差し替え可能な抽象（ポート）経由とし、フェイク実装だけでフロー全体をローカル検証できることを保証する。

## Boundary Context

- In scope: 監視フォルダ直下の処理対象検出と選別、OCR・構造化抽出・命名・リネーム・状態遷移のオーケストレーション、構造化抽出の出力スキーマ（信頼度・エビデンスを含む）とその命名入力への変換規則、低信頼度と技術エラーの分岐ポリシー、`app_llm_prompts/` の確定版プロンプト素材、フェイク実装とフェイク統合テスト。
- Out of scope: 実サービス（Drive API / Document AI / Gemini）への接続実装と認証（cloud-runtime-deploy が所有）、ホスト側テストブローカー（gcp-test-broker が所有）、インフラレベルのリトライ（ジョブ再試行・レート制限）、通知の送達（ログベースアラート構成は cloud-runtime-deploy）、命名規則そのもの（core-naming-engine が所有）、複数文書 PDF の分割。
- Adjacent expectations: core-naming-engine が定義する命名入力の形（文書日付・期間・タイトル・発行元）に合わせて本機能が抽出結果を変換する。本機能が定義するポート契約には、フェイク実装・テストブローカーのフィクスチャ・実アダプタの 3 者が従う（契約変更時は 3 スペック横断で再確認する）。

## Requirements

### Requirement 1: 処理対象ファイルの検出と選別

**Objective:** 運用者として、スキャナーがアップロードした未処理 PDF だけが自動処理され、それ以外のファイルには一切手を触れないでほしい。手動で整理したファイルや処理済み・要確認・エラーのファイルが勝手に変更されないようにするためである。

#### Acceptance Criteria

1. When リネームジョブが起動したとき, the ファイル検出処理 shall 監視対象フォルダ（`/From_BrotherDevice`）直下のファイル一覧を取得する。
2. When ファイル一覧を取得したとき, the ファイル検出処理 shall スキャナー生成名パターンに一致する未処理 PDF のみを処理対象として選別する。
3. If ファイルが処理済み（人間可読名）・要確認（`_needs_review_`）・エラー（`rename_error_`）・処理対象外のいずれかに分類されるとき, then the ファイル検出処理 shall そのファイルを変更せずスキップする。
4. The ファイル検出処理 shall 監視対象フォルダのサブフォルダ内のファイルを処理対象に含めない。
5. When 処理対象が 0 件のとき, the リネームパイプライン shall いかなるファイルも変更せず正常終了する。

### Requirement 2: 文書内容の構造化抽出

**Objective:** システム利用者として、スキャン PDF の内容から命名に必要なメタデータが信頼度と根拠付きで抽出されてほしい。抽出結果の確からしさに応じた自動判断を可能にするためである。

#### Acceptance Criteria

1. When 処理対象 PDF の内容を取得したとき, the 抽出パイプライン shall PDF から OCR テキストを取得する。
2. When OCR テキストを取得したとき, the 構造化抽出 shall 文書日付（元号表記の有無・元号文字列を含む）、書類タイトル、発行元、文書種別、期間（年分・年度分・明示期間）を構造化データとして抽出する。
3. The 構造化抽出 shall 主要フィールドごとに 0 から 1 の信頼度と短いエビデンス抜粋を付与する（OCR 全文をエビデンスに含めない）。
4. When 文書に複数の日付が含まれるとき, the 構造化抽出 shall 抽出ポリシーに定義された優先順位（発行日・作成日・交付日等）に従って文書日付を選択する。
5. If あるフィールドが文書から読み取れないとき, then the 構造化抽出 shall そのフィールドを欠落として表現する（推測値で埋めない）。
6. When ソース文書が元号表記で日付を記載しているとき, the 構造化抽出 shall 西暦に変換した日付とともに元号シグナル（元号表記の使用有無と元号文字列）を保持する。

### Requirement 3: 命名入力への変換とリネーム実行

**Objective:** システム利用者として、抽出メタデータが合意済み命名フォーマットのファイル名に変換され、Drive 上で実際にリネームされてほしい。Drive 上で文書を内容から探せるようにするためである。

#### Acceptance Criteria

1. When 構造化抽出結果の必須情報が十分な信頼度で得られたとき, the 命名変換処理 shall 抽出結果を命名エンジンの命名入力に変換し、合意済みフォーマットのファイル名を生成する。
2. If 文書日付が抽出できなかったか信頼度が閾値未満のとき, then the 命名変換処理 shall スキャナー生成名由来のスキャンタイムスタンプの日付を文書日付として使用する。
3. If 発行元または期間の信頼度が閾値未満のとき, then the 命名変換処理 shall 当該コンポーネントを省略してファイル名を生成する。
4. If 生成したファイル名が監視対象フォルダ内の既存ファイル名と重複するとき, then the リネーム処理 shall 重複解決済みの候補名（連番サフィックス付き）を使用する。
5. When 同一実行内で複数のファイルが同じ生成名に到達したとき, the リネーム処理 shall 同一実行内でもファイル名が重複しないよう重複解決を適用する。
6. When ファイル名が確定したとき, the リネーム処理 shall Drive 上の当該ファイルを確定名にリネームし、それ以外の変更を行わない。

### Requirement 4: 低信頼度時の要確認遷移

**Objective:** 運用者として、抽出が不確かな文書には誤った名前を付けず、要確認としてマークしてほしい。誤リネームによる文書の紛失を防ぎ、人間のレビューに委ねるためである。

#### Acceptance Criteria

1. If 書類タイトルが抽出できなかったか信頼度が閾値未満のとき, then the リネームパイプライン shall 最終名を生成せず、元のファイル名の先頭に `_needs_review_` を付与した名前にリネームする。
2. The リネームパイプライン shall 要確認判定に用いる信頼度閾値を設定値として保持し、コード本体を変更せずに調整できるようにする。
3. When 要確認遷移を実行したとき, the リネームパイプライン shall 対象ファイルと理由（不足・低信頼度のフィールドとその信頼度）を処理結果に記録する。
4. The リネームパイプライン shall 低信頼度による要確認を技術エラーと区別し、`rename_error_` を付与しない。

### Requirement 5: 技術エラー処理と障害の分離

**Objective:** 運用者として、1 つのファイルの技術的失敗が他のファイルの処理を止めず、失敗したファイルを Drive 上で識別・手動リトライできてほしい。バッチ全体の可用性と回復手段を確保するためである。

#### Acceptance Criteria

1. If OCR・構造化抽出・ファイル操作が技術的エラーで失敗したとき, then the リネームパイプライン shall 元のファイル名の先頭に `rename_error_` を付与した名前にリネームする。
2. If エラー状態へのリネーム自体が失敗したとき, then the リネームパイプライン shall 当該ファイルの失敗を処理結果に記録した上で、残りのファイルの処理を継続する。
3. While 複数のファイルを処理しているとき, the リネームパイプライン shall 1 ファイルの失敗（要確認・エラーのいずれも）によって他のファイルの処理を中断しない。
4. When ジョブ実行が完了したとき, the リネームパイプライン shall ファイルごとの処理結果（成功・要確認・エラー・スキップ）と理由の要約を呼び出し元に返す。
5. The リネームパイプライン shall 処理結果の記録に認証情報や OCR 全文を含めない（診断可能な要約のみを含める）。
6. When 運用者が要確認・エラーのファイル名を元のスキャナー生成名に戻したとき, the リネームパイプライン shall 次回実行でそのファイルを未処理として再処理する。

### Requirement 6: ランタイムプロンプト素材の確定

**Objective:** システム利用者（メンテナー）として、抽出・命名の挙動を自然言語の Markdown ポリシーを編集するだけで調整したい。コード変更なしに命名の好みを反映し続けられるようにするためである。

#### Acceptance Criteria

1. The 抽出パイプライン shall 構造化抽出への指示（日付の優先順位・元号の扱い・年分/年度分/期間の分類・エビデンスと信頼度の方針・構造化データの返し方）を `app_llm_prompts/` の確定版抽出ポリシーから読み込んで使用する。
2. The 抽出パイプライン shall 命名判断への指示（タイトルの選び方・発行元を含める条件・期間表記）を `app_llm_prompts/` の確定版命名ポリシーから読み込んで使用する。
3. When プロンプト素材ファイルが編集されたとき, the 抽出パイプライン shall コード変更なしに次回実行から更新後の内容を使用する。
4. If 確定版プロンプト素材が存在しないか読み込めないとき, then the 抽出パイプライン shall 暗黙のデフォルトで続行せず、明示的なエラーとして扱う。
5. The 確定版プロンプト素材 shall 機械的な安全規則（ファイル名サニタイズ・重複サフィックス・長さ制限・状態プレフィックス処理）を含めない（これらはコード側の責務とする）。

### Requirement 7: 外部サービスなしの検証可能性

**Objective:** 開発者として、Drive・OCR・LLM の実サービスに一切接続せずに、フロー全体をローカルで再現・検証したい。外部依存なしの高速で決定的なフィードバックループを確立するためである。

#### Acceptance Criteria

1. The リネームパイプライン shall 外部サービス（ファイル保管・OCR・構造化抽出）へのアクセスを差し替え可能な抽象経由でのみ行う。
2. The フェイク実装 shall 実サービスなしでファイル一覧・内容取得・リネーム、OCR、構造化抽出のふるまいを決定論的に再現する。
3. The フェイク実装 shall 実装差し替えのための抽象に対する契約（正常応答と失敗の表現）に従う。
4. When フェイク統合テスト（`integration_fake`）を実行したとき, the テストスイート shall フェイク実装のみでフロー全体（検出 → 抽出 → 命名 → リネーム → 状態遷移）を検証する。
5. When デフォルト設定でローカルテストを実行したとき, the テストスイート shall 外部サービスへ接続せずに完走する。
