# Implementation Plan

- [ ] 1. 基盤: Broker プロジェクトの初期化
- [ ] 1.1 `broker/` 配下に独立した uv プロジェクトを作成する
  - パッケージ本体の骨格（空のモジュール群）と、Broker 用の依存（FastAPI・uvicorn、開発用に pytest・httpx）を持つプロジェクト定義を作成する
  - ルートプロジェクトのランタイム依存・テスト収集設定に影響を与えないことを確認する（ルートのデフォルトテスト実行が従来どおり完走する）
  - `broker/` 内でテスト実行コマンドが成功する（0 件収集でも正常終了する）状態になっている
  - lint・型検査が `broker/` のコードにも適用されることを確認する
  - _Requirements: 1.4, 5.5_

- [ ] 2. Broker 本体: 設定・スキーマ・フィクスチャ解決・API
- [ ] 2.1 起動時設定の解決と検証を実装する
  - 環境変数からフィクスチャ格納場所（既定はリポジトリ外のホームディレクトリ配下）を解決し、`~` 展開を行う
  - 格納場所が存在しない・ディレクトリでない・読み取れない場合は原因と対象パスが分かる設定エラーで失敗する（曖昧なフォールバックをしない）
  - 単体テストで、未存在・ファイル指定・未設定時の既定値解決の各分岐が検証されている
  - _Requirements: 1.3, 4.4_
  - _Boundary: config_

- [ ] 2.2 (P) API 入出力スキーマを定義する
  - リクエスト（case_id）・OCR 応答・抽出応答・エラー応答の宣言的モデルを定義する
  - 抽出応答は上流スペック extraction-pipeline の抽出スキーマとフィールド 1:1 対応（欠落は null、日付は ISO 8601 文字列、期間は 3 種別の判別構造）とする
  - 信頼度の範囲（0.0–1.0）と期間種別ごとの必須フィールドをバリデータで検証し、違反データが検証エラーになることが単体テストで確認できる
  - 応答モデルに認証情報・ヘッダー・内部パスを表現するフィールドが存在しない
  - _Requirements: 2.2, 3.2, 5.3_
  - _Boundary: schemas_

- [ ] 2.3 case_id によるフィクスチャ解決を実装する
  - case_id を厳格なパターン（小文字英数字とアンダースコアのみ）で検証し、パス区切り・親ディレクトリ参照を含む値は解決を試みず拒否する
  - 許可リスト（フィクスチャ格納場所直下のケースディレクトリ集合）に対してのみ解決し、未知の case_id は「存在しない」ことだけが分かるエラーにする（内部パスを漏らさない）
  - ケースディレクトリから OCR フィクスチャと抽出フィクスチャを読み込み、スキーマ検証を通した値として返す。ファイル欠落・JSON 破損・スキーマ不適合は「フィクスチャ破損」として区別されたエラーにする
  - 単体テストで、正常解決・未知 case_id・不正形式（`../evil` や `a/b` 等）・破損 JSON の各分岐が検証されている
  - _Requirements: 2.1, 3.1, 4.1, 4.2, 4.3_
  - _Depends: 2.1, 2.2_
  - _Boundary: fixtures_

- [ ] 2.4 API エンドポイントとエラー応答を実装する
  - 稼働確認・OCR フィクスチャ・抽出フィクスチャの 3 エンドポイントのみを公開し、未定義パス（`/proxy` 等）が転送されずエラー応答になる
  - エラー応答を統一形式（形式不正 422 / 未知ケース 404 / フィクスチャ破損 500）とし、応答に内部パス・スタックトレースを含めない
  - 起動時に設定検証を実行し、不備時はプロセスが起動しない。アプリはテストから一時ディレクトリで構築できるファクトリ関数経由で生成する
  - ログはメソッド・パス・ステータス・case_id に限定し、リクエストヘッダーや環境変数を出力しない
  - テストクライアントによる単体テストで、正常応答（同一 case_id で同一応答が返ること）・全エラー分岐・未定義パスの拒否・応答に認証情報様の値が含まれないことが検証されている
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.3, 3.1, 3.3, 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Boundary: main_

- [ ] 3. 初期フィクスチャケース
- [ ] 3.1 サンプルフィクスチャ 4 ケースを作成する
  - 合意済み代表 3 文書例（住宅ローン残高証明書・確定申告書第一表・医療費通知）の OCR テキストと抽出結果の組を、上流の合意済み最終ファイル名に到達しうる内容（元号日付・年分/年度分期間・発行元を含む）で作成する
  - タイトル信頼度が既定閾値 0.7 未満の低信頼度ケースを 1 つ作成する
  - 各ケースの OCR テキストに抽出結果の全エビデンス文字列が含まれている（同一文書としての整合）
  - リポジトリ内の配布用サンプルディレクトリに 4 ケースが揃い、ケースレイアウト（ケースディレクトリ + OCR/抽出の JSON）が設計どおりである
  - _Requirements: 3.4, 7.1, 7.2, 7.3, 7.4_
  - _Depends: 2.2_
  - _Boundary: sample_fixtures_

- [ ] 3.2 サンプルフィクスチャの整合検証テストを追加する
  - サンプル全ケースがフィクスチャ解決層で読み込めること・スキーマ制約を満たすことを検証する
  - 各ケースのエビデンス文字列が OCR テキストに含まれること、低信頼度ケースのタイトル信頼度が閾値未満であることを検証する
  - Broker の単体テストスイートにこの検証が含まれ、全件グリーンで完走する
  - _Requirements: 7.1, 7.2, 7.3_
  - _Depends: 2.3, 3.1_

- [ ] 4. DevContainer からの到達確認とポート契約適合の検証
- [ ] 4.1 cloud マーカーテストの基盤を整備する
  - Broker 接続先を環境変数（既定は `host.docker.internal` 経由のホスト 8765 番）から解決するテストフィクスチャを作成する
  - テスト開始前に稼働確認エンドポイントへのプリフライトを行い、接続不能時は「Broker が未起動か到達できない」ことと起動手順の参照先を明示するメッセージで失敗する
  - 追加したテストはすべて `cloud` マーカーが付与され、デフォルトのローカルテスト実行では収集・実行されないことを確認する
  - _Requirements: 6.1, 6.2, 6.4_
  - _Boundary: tests/cloud_

- [ ] 4.2 フィクスチャ応答の契約適合テストを実装する
  - 前提: extraction-pipeline の実装（`scanner_rename` のポート契約・抽出スキーマ）がリポジトリに存在すること（未実装の場合は本タスクを保留し、上流実装後に実施する）
  - サンプル各ケースについて、OCR 応答から上流の OCR 結果型を、抽出応答から上流の抽出メタデータ型（不変条件検証付き）を構築できることを検証する。変換ヘルパはテスト内に閉じ、`src/` に共有変換モジュールを作らない
  - 未知の case_id に対するエラー応答（404 と統一エラーボディ）を検証する
  - テストコードが Broker 以外の外部サービスへ接続しない（Google API のホスト名がテストコードに現れない）
  - Broker をホストで起動した状態で `cloud` マーカーのテスト一式がグリーンになる
  - _Requirements: 6.1, 6.3, 6.5_
  - _Depends: 3.1, 4.1_
  - _Boundary: tests/cloud_

- [ ] 5. 運用手順の確定と全体検証

  - DevContainer 側の cloud テスト実行手順（環境変数・実行コマンド・Broker 起動が前提であること）をテスト README に追記する
  - 記載されたコマンドをそのまま実行して Broker が起動し、稼働確認エンドポイントが応答することを確認できる
  - _Requirements: 1.1, 1.4, 4.4, 7.4_

  - ルートのデフォルトテスト実行が cloud テストを収集せず従来どおり完走する
  - lint・型検査が Broker コードとルート側テストコードの双方で警告なしに通る
