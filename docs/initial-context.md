# 初期コンテキスト

このドキュメントは、プロジェクトの SDD 前の現時点での合意事項を記録します。

これは最終的な仕様書ではありません。

cc-sdd が正規の `spec`、`design`、`tasks` 成果物を生成した後は、それらの成果物が正となります。このドキュメントはインプット素材および歴史的コンテキストとして扱ってください。

## 背景

ユーザーは自宅に Brother スキャナーを持っています。スキャンした文書はスキャナーから直接 Google Drive にアップロードされます。

スキャナーと Google Drive はすでに連携済みで、アップロードは動作しています。

目標は、スキャンした PDF ファイルをその内容に基づいて自動的にリネームすることです。

ユーザーはこの小規模プロジェクトを、より大きなシステム開発で Claude Code を使う前の練習としても活用したいと考えています。

## スキャナーの既知の動作

スキャナー：

```text
Brother DCP-J987N / DCP-987N クラスデバイス
```

スキャナーは以下に PDF をアップロードします：

```text
Google Drive ルート /From_BrotherDevice
```

スキャナーが生成するファイル名は以下のパターンに従います：

```regex
^\d{14}_\d{3}\.pdf$
```

例：

```text
20260507132742_001.pdf
```

タイムスタンプ部分は以下のように解釈します：

```text
YYYYMMDDhhmmss
```

サフィックスは複数のスキャンファイルに対する3桁の連番です。

## ランタイムプラットフォーム

選択したクラウドプラットフォームは Google Cloud です。

現在想定しているランタイムアーキテクチャ：

```text
Cloud Scheduler
  -> Cloud Run Job
    -> Google Drive API
    -> Document AI OCR
    -> Gemini 構造化出力
    -> Google Drive リネーム
    -> Cloud Logging / Cloud Monitoring
```

## トリガー

v1 ではポーリングを使用します。

ジョブは5分ごとに実行し、`/From_BrotherDevice` の直下を検索します。

Google Drive プッシュ通知は v1 の対象外です。

## 対象ファイル

`/From_BrotherDevice` の直下のみを対象とします。

スキャナー生成パターンに一致する名前の PDF のみを対象とします：

```regex
^\d{14}_\d{3}\.pdf$
```

このパターンに一致しないファイル名のものは、処理済み、スキップ済み、要確認、または失敗として扱います。

## 状態管理

v1 では Firestore や他の外部ステートストアを使用しません。

キューの状態は Google Drive のファイル名で表現します。

### 成功

成功した場合、ファイルは人間が読める名前にリネームされ、スキャナー生成キューから自然に外れます。

例：

```text
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書.pdf
```

### 低信頼度

OCR/LLM の抽出が不確かな場合、元のファイル名の先頭に以下を付与します：

```text
_needs_review_
```

例：

```text
_needs_review_20260507132742_001.pdf
```

### エラー

技術的なエラーで処理が失敗した場合、元のファイル名の先頭に以下を付与します：

```text
rename_error_
```

例：

```text
rename_error_20260507132742_001.pdf
```

## 手動リトライ

ファイル名を元のスキャナー生成名に戻すことで手動リトライが可能です。

例：

```text
rename_error_20260507132742_001.pdf
```

を以下に戻す：

```text
20260507132742_001.pdf
```

## PDF 解析

現在想定しているメインパイプライン：

```text
PDF
  -> Document AI OCR
  -> OCR テキストとレイアウト
  -> Gemini 構造化出力
  -> 抽出メタデータ
  -> 命名エンジン
```

Gemini による PDF 直接解析を代替案としてテストする場合もありますが、メインの方向は Document AI OCR の後に Gemini 抽出を行うことです（障害の診断がしやすいため）。

## 抽出するメタデータ

アプリケーションは以下の抽出を試みます：

- 文書日付
- ソースで日本の元号表記が使われているかどうか
- 該当する場合の元号文字列
- 文書タイトル
- 発行者・組織・銀行名
- 文書種別
- `2025年分` などの年ベースの期間
- `2025年度分` などの年度ベースの期間
- `202504-202603` などの明示的な期間
- 信頼度の値
- エビデンスの抜粋

## 基本的な命名フォーマット

大まかな命名フォーマット：

```text
<文書日付>_<年分・年度分・期間補足>_<書類タイトル>[_発行元].pdf
```

該当しない場合は期間コンポーネントを省略します。

有用でない場合や信頼性の高い抽出ができない場合は発行者コンポーネントを省略します。

## 日付と元号表記

文書の日付が日本の元号表記で記載されている場合、西暦と元号の両方を含めます。

例：

```text
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書.pdf
```

文書の日付が西暦のみで記載されている場合は、元号表記を強制しません。

## 年分・年度分の命名

ファイル名に `対象` という言葉を使わないでください。

ユーザーは以下のフォーマットを好みます。

### 年度ベースの文書

例：

```text
20260415(R8)_2025年度分(202504-202603)_医療費通知.pdf
```

### 所得税・確定申告の年分文書

例：

```text
20260310(R8)_2025年分_確定申告書第一表.pdf
```

## 発行者・銀行名

発行者は常に必要というわけではありません。

ファイルを区別するためにタイトルだけでは不十分な場合に発行者を含めることがあります。

ユーザーが複数の銀行ローンを持つ可能性があることから、住宅ローン関連文書では特に重要です。

例：

```text
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書_三井住友銀行.pdf
```

タイトルだけで十分な場合はノイズになるため発行者を含めません。

多くの官公庁文書では、タイトルと年分・年度分情報だけで十分な場合があります。

正確なポリシーは、メンテナンスが難しい YAML ルールセットではなく、自然言語のランタイム LLM プロンプト素材として管理します。

## 1つのPDF = 1つの文書

1つのPDFを1つの論理的な文書として扱います。

ユーザーは手動でスキャン単位を管理します。

v1 では複数文書のPDFを分割しません。

## ファイル名の重複

Google Drive は同名ファイルを保持できますが、アプリケーションは人間が混乱するような重複を避けます。

生成した名前がすでにフォルダに存在する場合は、サフィックスを追加します。

例：

```text
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書.pdf
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書_2.pdf
20211001(R3)_住宅取得資金に係る借入金の年末残高等証明書_3.pdf
```

## 元ファイルの保存

v1 では元ファイルの別途バックアップは不要です。

リネームのみ行います。

## 通知

`needs_review` と `error` のみ通知が必要です。

成功時は通知不要です。

v1 で好むアプローチ：

```text
構造化アプリケーションログ
  -> Cloud Logging
  -> Cloud Monitoring ログベースアラート
  -> メール通知
```

通知先：

```text
fumiaki.k@gmail.com
```

v1 では Gmail API を直接使った送信は実装しません。

## ローカル開発

ローカル開発環境には DevContainer を使用します。

Claude Code は DevContainer の中で実行します。

GCP Test Broker は Mac ホスト上で、DevContainer の外で実行します。

GCP 認証情報（ADC）は named volume 経由でコンテナ内に存在します。Claude Code による読み取りは settings.json の deny リストと規範ルールで抑止しますが、これはベストエフォートの措置であり完全な技術的強制ではありません。コンテナ内への ADC 存在は受容済みリスクとして管理します。詳細は `docs/security-notes.md` を参照してください。

## クラウド統合テスト

ユーザーは Claude Code にクラウド統合テストを実行させ、失敗を自己修正させたいと考えています。

ただし、Claude Code は生の GCP 認証情報を読み取ったり受け取ったりしてはいけません。

したがって、クラウド統合テストはローカルのホスト側 GCP Test Broker を呼び出す必要があります。

ブローカーは Google Cloud API を呼び出します。

DevContainer は Google Cloud API を直接呼び出してはいけません。
