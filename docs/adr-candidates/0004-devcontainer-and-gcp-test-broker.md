# ADR候補 0004: DevContainer とホスト側 GCP Test Broker の使用

## ステータス

候補。

## コンテキスト

ユーザーは Claude Code にクラウド統合テストを実行させ、失敗を自己修正させたいと考えています。

同時に、Claude Code は GCP 認証情報を読み取ったり漏洩させたりしてはいけません。

DevContainer に ADC、サービスアカウントキー、またはリフレッシュトークンへのアクセスを直接与えると、認証情報の漏洩リスクが高まります。

## 候補の決定

Claude Code と通常の開発ツールを DevContainer 内で実行します。

GCP Test Broker を Mac ホスト上で実行します。

GCP 認証情報はホスト上のみに保持します。

DevContainer 内のクラウド統合テストは Google Cloud API に直接アクセスせず、ブローカーを呼び出します。

## 想定するトポロジー

```text
Mac ホスト
  ├─ ~/.config/gcloud/
  ├─ GCP Test Broker
  │   └─ 127.0.0.1:8765
  │
  └─ DevContainer
       ├─ Claude Code
       ├─ pytest
       └─ クラウドテスト -> ブローカー
```

## 根拠

これにより、DevContainer に生の認証情報を公開することなく、Claude Code がクラウド統合テストを実行できます。

ブローカーは目的を絞ったテスト API のみを公開し、ログからシークレットを除去できます。

認証情報リスクを低減しながら、望ましい Claude Code 開発ワークフローを維持できます。

## 結果

ポジティブ：

- Claude Code がクラウドテストを実行できる
- Claude Code が失敗を修正できる
- GCP 認証情報が DevContainer に入らない
- DevContainer により環境の再現性が向上
- ブローカー API で許可リスト化された操作とフィクスチャを強制できる

ネガティブ：

- 動作する部品が増える
- ブローカーのメンテナンスが必要
- DevContainer とホスト間のネットワーク設定が必要
- セキュリティはブローカー API が狭く非汎用であることに依存

## ブローカーの制約

ブローカーは以下を公開してはいけません：

- 生のアクセストークン
- リフレッシュトークン
- サービスアカウント JSON
- 任意の Google API プロキシ
- 任意のローカルファイルアップロード

クラウドテストフィクスチャは `case_id` で参照します。

## 未解決事項

- ブローカーの実装言語とフレームワーク
- 最小限の v0 ブローカーエンドポイント
- 正確な DevContainer のネットワーク制限
- v1 でのエグレスファイアウォールの厳密さ
