# ADR候補 0003: v1 では外部ステートストアなし

## ステータス

候補。

## コンテキスト

スキャナー生成ファイルには予測可能な命名パターンがあります：

```regex
^\d{14}_\d{3}\.pdf$
```

ファイルのリネームに成功すると、スキャナー生成パターンに一致しなくなります。

したがって、ファイル名自体でキューの状態を表現できます。

## 候補の決定

v1 では Firestore や他の外部ステートストアを使用しません。

スキャナー生成パターンに一致するファイルを未処理のキューアイテムとして扱います。

リネームによって終端状態を表現します：

```text
成功：
  <生成された人間が読める名前>.pdf

低信頼度：
  _needs_review_<元のファイル名>.pdf

技術的エラー：
  rename_error_<元のファイル名>.pdf
```

## 根拠

ステートストアを維持するよりもシンプルです。

v1 の主な運用上のニーズは、無限の再処理を避けることとファイルを失わないことです。

プレフィックスベースの終端状態はそのニーズを満たします。

## 結果

ポジティブ：

- Firestore のセットアップが不要
- 権限が少ない
- 実装の複雑さが少ない
- 元のスキャナー生成ファイル名を復元することで簡単な手動リトライが可能

ネガティブ：

- 詳細な処理履歴がない
- 試行回数のカウンターがない
- ログ以外の構造化された監査証跡がない
- 手動リトライはリネーム操作
- 後でレビュー UI を構築するのが難しい

## 将来の選択肢

レビューワークフロー、監査履歴、試行回数、またはダッシュボードが必要になった場合は、v2 で Firestore や他のステートストアを追加します。
