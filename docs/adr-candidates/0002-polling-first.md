# ADR候補 0002: Google Drive プッシュ通知の代わりにポーリング優先

## ステータス

候補。

## コンテキスト

スキャナーが Google Drive に PDF をアップロードした後、リネームをトリガーする必要があります。

考えられるアプローチ：

- Google Drive プッシュ通知
- スケジュールによるポーリング

## 候補の決定

v1 ではスケジュールポーリングを使用します。

5分ごとに実行します。

v1 では Google Drive プッシュ通知を使用しません。

## 根拠

ポーリングの方が実装とデバッグが簡単です。

この個人の文書スキャンワークフローでは、最大5分程度の遅延は許容できます。

Google Drive プッシュ通知は追加の複雑さをもたらします：

- Webhook エンドポイント
- 通知チャンネルのライフサイクル
- チャンネルの更新
- 変更一覧の取得
- 重複通知
- テストすべき動作する部品が増える

## 結果

ポジティブ：

- v1 がシンプル
- ローカルテストが容易
- 障害からの復旧が容易
- 本番コンポーネントが少ない

ネガティブ：

- リネームが即時ではない
- 繰り返しのポーリングで多少の API クォータを消費
- ほぼリアルタイムの処理が必要になった場合はプッシュ通知を再検討する可能性あり

## 未解決事項

- 実際の使用後も5分が適切なインターバルかどうか
- まだアップロード中のファイルの読み取りを避けるために、最後に更新されてから N 秒以内のファイルをスキップすべきかどうか
