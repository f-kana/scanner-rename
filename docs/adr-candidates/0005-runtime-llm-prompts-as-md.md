# ADR候補 0005: ランタイム LLM プロンプトを Markdown で管理

## ステータス

候補。

## コンテキスト

システムは LLM（おそらく Gemini）を使用して、OCR 結果または PDF から文書メタデータを抽出します。

ユーザーは命名の挙動を簡単にメンテナンスできるようにしたいと考えています。

以前検討した `naming_rules.yaml` はユーザーがメンテナンスするには難しすぎると判断されました。

`prompts/` という用語は曖昧で、以下の意味に解釈される可能性があります：

- Claude Code プロンプト
- cc-sdd プロンプト
- ランタイムの Gemini プロンプト
- ユーザー向けプロンプトスニペット

## 候補の決定

アプリケーションのランタイム LLM プロンプトは以下に保存します：

```text
app_llm_prompts/
```

人間がメンテナンスできるポリシーと例を Markdown ファイルで管理します。

初期ファイル：

```text
app_llm_prompts/
  README.md
  naming_policy.draft.md
  extraction_policy.draft.md
```

## 根拠

Markdown は深い構造の YAML よりもユーザーがメンテナンスしやすいです。

`app_llm_prompts/` というディレクトリ名により、これらのプロンプトが Claude Code ではなくアプリケーションのランタイム LLM 向けであることが明確になります。

## 結果

ポジティブ：

- 人間が読めてメンテナンスしやすい
- 例の追加が容易
- Gemini の挙動の調整が容易
- Claude Code の指示とアプリケーション LLM プロンプトの混同を避けられる

ネガティブ：

- 厳密な設定スキーマほど決定論的ではない
- 良いテストとゴールデンケースが必要
- 一部の機械的なルールはコードに残す必要がある

## 境界

ランタイム LLM プロンプトがカバーすべき内容：

- 文書日付の識別方法
- 元号表記の識別方法
- タイトルの識別方法
- 発行者・銀行名を含める場合
- 年分・年度分・期間情報の分類方法
- 構造化 JSON の返し方

コードがカバーすべき内容：

- ファイル名のサニタイズ
- 重複時のサフィックス付与
- 最大長の処理
- スキャナーファイル名パターンのマッチング
- `_needs_review_` や `rename_error_` などの終端プレフィックス
